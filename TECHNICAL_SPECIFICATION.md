# Техническое задание: PWA-платформа координации студенческих мероприятий

## 1. Общие сведения

| Параметр | Значение |
|----------|----------|
| **Наименование системы** | PWA-платформа координации студенческих мероприятий |
| **Тип приложения** | Progressive Web App (SPA + Service Worker + Manifest) |
| **Целевая аудитория** | Студенты, преподаватели-организаторы, администрация вуза, гости |
| **Контекст использования** | Внутренняя экосистема одного конкретного вуза |
| **Базовая архитектура** | Трёхуровневая клиент-серверная: PWA Client ↔ REST API Server ↔ СУБД + Кэш + Хранилище |
| **Язык интерфейса** | Русский (с поддержкой i18n на будущее) |

## 3. Технологический стек (утверждённый)

| Уровень | Технология | Назначение |
|---------|------------|------------|
| **Frontend (PWA)** | React 18+ + Vite + TypeScript + @tanstack/react-query | SPA-ядро, state-management, умное кэширование, offline-реактивность |
| **UI-библиотека** | Material UI (MUI) v5+ | Полное соответствие Material Design 3 |
| **Формы и валидация** | React Hook Form + Zod | Валидация на клиенте |
| **PWA-инфраструктура** | vite-plugin-pwa + Workbox + IndexedDB (idb) | Манифест, Service Worker, фоновая синхронизация |
| **Карты** | Яндекс.Карты API (JS API 3.0) | Отображение адресов мероприятий, координаты |
| **Backend (API)** | NestJS + TypeScript | REST API, валидация, аутентификация, бизнес-логика |
| **СУБД** | PostgreSQL 15+ + Prisma ORM | Реляционное хранение, транзакции, JSONB для гибких полей |
| **Кэш & Очереди** | Redis + BullMQ | Сессии, rate-limit, кэш афиши, асинхронные задачи |
| **Хранение медиа** | S3-совместимое (Cloudflare R2 / MinIO) + CDN | Файлы хранятся отдельно, в БД только URL-ссылки |
| **DevOps** | Docker, Nginx, GitHub Actions, Sentry | CI/CD, мониторинг, логирование, контейнеризация |

## 4. Ролевая модель и матрица доступа (RBAC)

| Роль | Доступные действия | Ограничения |
|------|-------------------|-------------|
| **Гость** | Просмотр публичной афиши, использование фильтров (тип, дата, институт), переход на страницы входа/регистрации | Нет ЛК, нельзя записываться на события, оставлять сообщения или создавать мероприятия |
| **Студент** | Всё как у Гостя + ЛК, указание института/группы, запись на события, портфолио посещений, экспорт в календарь, просмотр/скачивание сертификатов | Не может создавать/редактировать мероприятия, модерация недоступна |
| **Организатор** | Всё как у Студента + CRUD своих мероприятий, загрузка медиа, указание контактов и ссылки на чат, генерация QR для чек-ина, базовая аналитика по своим событиям, **ручная загрузка сертификатов** для участников | Доступ только к своим данным, требует модерации новых событий |
| **Администратор** | Полный контроль: модерация очереди, управление пользователями/ролями, глобальная аналитика по вузу, настройки платформы, блокировки, экспорт отчётов | Доступ через отдельный роут /admin, не виден в пользовательском интерфейсе |

## 5. Функциональные требования (по модулям)

### 5.1. Модуль аутентификации и пользователей

- **Регистрация/вход:** login+password (email/username) с валидацией уникальности
- **Сброс пароля:** через email (ссылка с TTL 1 час, верификация почты обязательна)
- **Токены:** JWT Access (15 мин) + Refresh (7 дней) с ротацией и отзывом при выходе
- **Профиль студента** (адаптировано под один вуз):
  - ФИО, дата рождения, телефон, email, фото (URL)
  - `institute` (подразделение/институт вуза, выпадающий список из справочника)
  - `group` (учебная группа, текстовое поле с валидацией формата)
  - `year_of_study` (1–6), `status` (учится/отчислен/выпускник)
  - **История посещённых мероприятий** (с датой и статусом участия)
  - **Сертификаты** (список загруженных организаторами файлов для скачивания)
- **API:** 
  - `POST /auth/register`
  - `POST /auth/login`
  - `POST /auth/logout`
  - `GET/PUT /users/me`
  - `GET /users/me/certificates`
  - `GET /users/me/events-history`

### 5.2. Модуль мероприятий (Каталог + CRUD + Коммуникация)

- **Карточка мероприятия:**
  - название, тип, организатор, дата/время, адрес, **координаты (Яндекс.Карты)**, описание
  - лимит мест, статус
  - `contacts` (email/телефон ответственного)
  - `chat_link` (URL на Telegram/VK/Дискорд чат мероприятия)
- **Коммуникация:** Ссылка на внешний чат открывается во внешней вкладке/приложении
- **Фильтрация:** по типу, дате (от сегодня), институту, статусу
- **Пагинация,** кэширование на 5 мин, сортировка по дате
- **API:**
  - `GET /events`
  - `GET /events/:id`
  - `POST /events`
  - `PUT /events/:id`
  - `DELETE /events/:id`

### 5.3. Модуль регистрации и учёта посещаемости

- **RSVP:** `POST /events/:id/register` → запись в `attendees`
- **Отмена:** `DELETE /events/:id/register` (до дедлайна или за 2ч до начала)
- **Чек-ин:** динамический QR (обновляется каждые 30 сек), сканирование камерой PWA → `POST /attendance/check`
- **История посещений** доступна в ЛК студента
- **API:**
  - `POST /attendance/scan`
  - `GET /events/:id/attendees`

### 5.4. Административный модуль

- **Очередь модерации:** список `status=на модерации`, кнопки опубликовать/отклонить
- **Управление пользователями:** блокировка, смена роли, сброс сессий
- **Аналитика:** посещаемость по институтам/группам, активность объединений, экспорт CSV/PDF
- **Справочники:** типы мероприятий, список институтов/факультетов вуза
- **API:**
  - `GET /admin/moderation`
  - `PUT /admin/users/:id`
  - `GET /admin/analytics`

### 5.5. Уведомления и напоминания

- **Push через VAPID:** за 24ч, за 2ч, отмена, изменение времени/места *(работает на Android и iOS 16.4+ при установленном PWA)*
- **Подписка/отписка** в настройках профиля
- **Фоновая синхронизация:** регистрация/чек-ин сохраняется в IndexedDB при офлайне и отправляется при восстановлении связи
- **Экспорт в календарь:** генерация `.ics` файла + `webcal://` ссылка

### 5.6. Модуль сертификатов

- **Загрузка организатором:** после завершения мероприятия организатор загружает PDF-сертификаты для участников (массово или индивидуально)
- **Хранение:** файлы в S3, в БД только URL + связь с пользователем и мероприятием
- **Просмотр/скачивание:** студент видит список доступных сертификатов в ЛК
- **API:**
  - `POST /events/:id/certificates/upload` (Организатор)
  - `GET /events/:id/certificates` (Организатор — свои)
  - `GET /users/me/certificates` (Студент — свои)

## 6. Требования к данным и структуре БД

| Сущность | Ключевые поля | Индексы | Комментарии |
|----------|---------------|---------|-------------|
| `users` | id, role, login, email, password_hash, status, created_at | UNIQUE(login), UNIQUE(email) | Общая таблица для всех ролей |
| `student_profiles` | user_id, institute, group, year_of_study, status, birth_date, phone, avatar_url | user_id | 1:1 с users, адаптировано под 1 вуз |
| `events` | id, organizer_id, type, name, datetime, address, lat, lng, status, limit, contacts, chat_link | datetime, status, organizer_id, institute | contacts (JSON/TEXT), chat_link (URL), **lat/lng DECIMAL(9,6)** |
| `event_images` | id, event_id, url, order, created_at | event_id | Хранится в S3, в БД только URL |
| `registrations` | id, event_id, user_id, status, registered_at, checked_in_at | (event_id, user_id) UNIQUE | RSVP + чек-ин |
| `certificates` | id, event_id, user_id, file_url, uploaded_at, uploaded_by | (event_id, user_id) UNIQUE | **Новая сущность**, файл в S3 |
| `push_subscriptions` | id, user_id, endpoint, p256dh, auth, created_at | user_id | Для Web Push |
| `institutes_dict` | id, name, code, is_active | name | Справочник подразделений вуза |

### ⚠️ Критично:
- **Запрещено хранить изображения и сертификаты в BLOB.** Только URL-ссылки на S3.
- Все пароли хэшируются **bcrypt/argon2**.
- Использовать **soft delete** (`is_deleted BOOLEAN`) для аудита.
- Координаты в **DECIMAL(9,6)**.

## 7. Нефункциональные требования

| Параметр | Требование |
|----------|------------|
| **Производительность** | TTFB < 800 мс, FCP < 3 сек, LCP < 2.5 сек. Кэширование статики CDN, API-ответов Redis (TTL 300 сек) |
| **Масштабируемость** | Stateless API, горизонтальное масштабирование, read-replicas БД при > 2000 RPS |
| **Отказоустойчивость** | Graceful degradation при падении БД (отдаётся кэш), retry-логика на клиенте |
| **PWA-стандарты** | Lighthouse PWA > 90, установка на iOS/Android, offline-fallback, фоновая синхронизация |
| **Локализация** | Формат дат/времени по `Intl.DateTimeFormat`, поддержка часовых поясов (UTC в БД) |
| **Доступность (a11y)** | WCAG 2.1 AA: контраст ≥ 4.5:1, навигация с клавиатуры, aria-* атрибуты |
| **Push-уведомления** | Работают на Android и iOS 16.4+. На iOS ниже 16.4 уведомления могут не приходить — это допустимо |

## 8. Требования к UI/UX и дизайну

- **Система:** Material Design 3 (Material UI)
- **Адаптивность:** Mobile-first, breakpoints: 320px, 768px, 1024px, 1440px
- **Состояния загрузки:** Skeletons, индикаторы офлайн-режима, пустые состояния (empty states)
- **Навигация:** Header (лого, поиск, меню), Bottom Tab (мобильные), Footer (контакты вуза, политика)
- **Обработка ошибок:** Toast-уведомления, human-readable сообщения, retry-кнопки
- **Темизация:** Светлая/тёмная тема, системный `prefers-color-scheme`
- **Карты:** Интеграция Яндекс.Карт для отображения адреса мероприятия с возможностью открыть в приложении Яндекс.Карт

## 9. Требования к безопасности

| Вектор | Защита |
|--------|--------|
| **Аутентификация** | JWT с коротким TTL, Refresh Token Rotation, secure/httpOnly cookies (опционально) |
| **Авторизация** | RBAC на уровне API-шлюза, middleware проверки ролей, ownership-check для CRUD |
| **Ввод данных** | Валидация на клиенте (Zod/Yup) и сервере, санитизация, CSP headers |
| **Сетевая защита** | HTTPS (TLS 1.3), CORS whitelist, rate-limit (IP + user), Secure headers |
| **Уязвимости** | Защита от XSS, SQLi, CSRF, brute-force (lockout после 5 попыток) |
| **Бэкапы** | DB: ежедневно + инкрементальные. S3: версионирование. Хранение 30 дней |

---

## 10. Исключения и ограничения

- **Интеграция с системами вуза** (LMS, LDAP, SAML) — не требуется на текущем этапе
- **Навыки в профиле студента** — исключены из требований
- **Push-уведомления на iOS < 16.4** — могут не работать, это допустимое ограничение
- **Генерация сертификатов** — только ручная загрузка организатором, автоматическая генерация не предусмотрена

## 11. План разработки (рекомендованный)

### Этап 1: Ядро (4-6 недель)
- Настройка инфраструктуры (Docker, БД, Redis, S3)
- Модуль аутентификации и пользователей
- Базовый CRUD мероприятий
- Интеграция Яндекс.Карт

### Этап 2: Участие и посещаемость (3-4 недели)
- Регистрация на мероприятия (RSVP)
- Система чек-ина (QR-коды)
- История посещений в ЛК

### Этап 3: Администрирование и сертификаты (3-4 недели)
- Панель модерации
- Управление пользователями
- Загрузка и просмотр сертификатов

### Этап 4: PWA и уведомления (2-3 недели)
- Service Worker, offline-режим
- Push-уведомления
- Экспорт в календарь

### Этап 5: Тестирование и запуск (2-3 недели)
- Нагрузочное тестирование (k6)
- Аудит безопасности
- Пилотный запуск в вузе
